# Portfolio Pin Criteria

기준일: 2026-04-06

이 문서는 어떤 프로젝트를 GitHub pin에 올리고 내릴지 판단하는 내부 기준이다.

## 기본 원칙

- pin은 "좋아하는 프로젝트"가 아니라 "현재 포트폴리오 서사를 가장 잘 설명하는 프로젝트"여야 한다.
- 하나의 프로젝트는 적어도 하나의 강한 메시지를 증명해야 한다.
- README가 약하거나 결과 예시가 없으면 완성도가 높아도 pin 효율이 떨어진다.
- 아직 구현 전 단계인 프로젝트라도 문제 정의와 사용자 흐름이 매우 선명하면 후보가 될 수 있다. 다만 이 경우 공개 문구에서 과장하지 않는다.

## 공통 통과 조건

- 저장소 첫 화면에서 문제와 해결 방향이 10초 안에 읽힌다.
- README에 현재 상태가 명확히 적혀 있다.
- Quick start 또는 실행 방법이 있다.
- 최소 1개의 결과 예시가 있다.
  - 스크린샷
  - 다이어그램
  - API 예시 응답
  - 입력/출력 샘플
- 저장소 설명이 기술 목록만 나열하지 않는다.
- 최근 손봤거나, 앞으로도 포트폴리오 핵심 축으로 유지할 이유가 있다.

## 카테고리별 추가 기준

### Backend / Data / Infra

- 아키텍처 설명이 있다.
- 테스트 또는 검증 결과가 있다.
- 데이터 흐름이나 운영 포인트가 드러난다.
- 단순 CRUD 저장소처럼 보이지 않는다.

### AI / Product / Marketing

- 사용자 문제 정의가 분명하다.
- 입력에서 결과까지의 흐름이 보인다.
- 모델/규칙 기반 분석이라면 한계와 현재 상태를 정직하게 적는다.
- 제품 포지셔닝 또는 전환 흐름이 어느 정도 정리돼 있다.

### Workflow / Integration Tool

- 누가 왜 쓰는지 바로 설명된다.
- 설정 문서가 지나치게 길지 않다.
- 실패 처리나 예외 상황에 대한 설명이 있다.
- 실제 사용 흐름을 보여주는 캡처나 예시가 있다.

## 기본 pin 구성 규칙

- 2개: 백엔드 정체성
  - 기본값: `asset-radar`, `card-mizer`
- 2개: AI / 제품 / 마케팅 확장
  - 기본값: `SignalMate`, `blog-to-shorts`
- 2개: 실제 사용자 문제 해결 도구
  - 기본값: `Session Pilot`, `discord-kakao-translator`

## 교체 조건

- README와 공개 설명이 현재 포트폴리오 방향과 어긋난다.
- 저장소가 오래 방치돼 대표성이 약해졌다.
- 더 넓은 분야를 더 잘 보여주는 후보가 생겼다.
- 같은 메시지를 더 강하게 증명하는 프로젝트가 생겨 중복도가 높아졌다.

## 유지 조건

- 현재 pin 한 칸을 차지할 이유를 한 문장으로 설명할 수 있다.
- 같은 카테고리 후보와 비교했을 때 대표성이 여전히 높다.
- 포트폴리오 페이지, GitHub pin, README가 같은 방향을 가리킨다.

## 대체 후보 메모

- `meetric`: 대중적인 웹 제품 감각이 필요할 때
- `webhook-notify`: SDK/라이브러리 축을 강조할 때
- `mbti-self-cdm`: 설계 문서와 모바일 확장을 다시 밀고 싶을 때
