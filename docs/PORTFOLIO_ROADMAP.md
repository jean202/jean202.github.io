# Portfolio Roadmap

기준일: 2026-04-06

이 문서는 포트폴리오를 다음 단계로 끌어올릴 때 무엇부터 손볼지 정리한 내부 작업 계획서다.

## 전체 우선순위

1. 공개 표면 정리: 포트폴리오 페이지, GitHub pin, README 첫 문장 정렬
2. 백엔드 대표작 완성도 상승: `asset-radar`, `card-mizer`
3. AI/제품 축 고도화: `SignalMate`, `blog-to-shorts`
4. 워크플로우/도구 축 고도화: `Session Pilot`, `discord-kakao-translator`

## Phase 1. Public Surface

- 포트폴리오 페이지에서 대표 6개 서사를 고정한다.
- GitHub pin 제목/설명 문구를 새 서사에 맞게 맞춘다.
- 각 저장소 README 첫 3문단을 문제, 해결 방식, 무엇을 증명하는지 중심으로 정리한다.
- 프로젝트별 대표 스크린샷 또는 다이어그램을 최소 1개씩 준비한다.

## Project Work Queue

### asset-radar

- 포트폴리오 역할: 백엔드 + 데이터 + 인프라를 동시에 보여주는 핵심 축
- 목표 상태: "실시간 데이터 제품"으로 읽히는 저장소
- 다음 작업:
  - README 예시 응답과 대시보드 스크린샷 정리
  - 아키텍처 다이어그램과 데이터 소스 설명 보강
  - 운영/배포 분리와 환경 설명 정리

### card-mizer

- 포트폴리오 역할: 구조 설계와 도메인 모델링 증명
- 목표 상태: "잘 설계된 백엔드 데모"로 바로 이해되는 저장소
- 다음 작업:
  - 데모 UI 스크린샷과 주요 시나리오 정리
  - 추천 로직의 핵심 판단 기준을 문서로 압축
  - PostgreSQL/Testcontainers 쪽 신뢰도 보강

### SignalMate

- 포트폴리오 역할: AI + 제품 + UX + 전환 설계
- 목표 상태: "AI 기능이 들어간 소비자용 제품 프로토타입"으로 읽히는 저장소
- 다음 작업:
  - 랜딩과 분석 화면을 하나의 사용자 여정으로 묶기
  - 규칙 기반 분석 결과 예시와 근거 카드 캡처 추가
  - 저장, 결제, 개인정보 처리 정책의 MVP 수준 설명 보강

### blog-to-shorts

- 포트폴리오 역할: AI 자동화 + 마케팅/성장 실험
- 목표 상태: "콘텐츠 제작 자동화 파이프라인"으로 설득력 있는 저장소
- 다음 작업:
  - 입력 블로그 글에서 최종 Shorts까지 결과 예시 추가
  - 실패 케이스와 재시도 전략 문서화
  - 업로드 이력 또는 결과 메타데이터 화면 정리

### Session Pilot

- 포트폴리오 역할: AI workflow 문제를 푸는 제품 축
- 목표 상태: "AI 작업 인박스" 문제 정의가 선명한 프로토타입
- 다음 작업:
  - PRD와 상업화 재정의 문서를 한 화면 흐름으로 연결
  - 핵심 대시보드 와이어프레임 또는 프로토타입 제작
  - 자동 감지, 검토 큐, 재진입 요약 중 최소 1개 실제 동작 흐름 구현

### discord-kakao-translator

- 포트폴리오 역할: 실제 생활 문제를 끝까지 해결하는 통합 도구
- 목표 상태: "설치하면 바로 쓰는 개인 생산성 브리지"로 읽히는 저장소
- 다음 작업:
  - 상태 UI와 트레이 앱 캡처 추가
  - 설정 흐름을 더 짧게 만드는 온보딩 문서 정리
  - 에러/억제/재시도 사례를 README에서 더 명확히 설명

## 공통 체크리스트

- 프로젝트의 한 줄 정의가 README 첫 문단에 있는가
- 저장소를 열자마자 문제와 결과가 보이는가
- 스크린샷, 다이어그램, 예시 응답 중 최소 하나가 있는가
- 테스트나 검증 결과를 한 줄로 보여주는가
- "현재 상태"와 "다음 단계"가 정직하게 적혀 있는가

## 교체 후보 관리

- `meetric`: 일반 사용자용 웹 제품 감각이 필요할 때 대체 후보
- `webhook-notify`: 라이브러리/SDK 역량을 밀고 싶을 때 대체 후보
- `mbti-self-cdm`: 설계 문서와 모바일 확장 흐름을 다시 살리고 싶을 때 재검토
