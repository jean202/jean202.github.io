# Portfolio Strategy

기준일: 2026-04-06

이 문서는 `jean202.github.io` 포트폴리오 페이지, GitHub pin, 각 저장소 README를 하나의 서사로 맞추기 위한 내부용 전략 문서다. `docs/` 폴더는 Jekyll 빌드에서 제외되므로 GitHub Pages 블로그에는 노출되지 않는다.

## 공개 서사 핵심 문장

백엔드를 중심으로 일하면서 AI, 데이터 분석, UI/UX, 인프라, 마케팅 등 제품을 이루는 다양한 분야에 관심을 두고 있습니다.

## 포트폴리오가 보여줘야 하는 흐름

1. `asset-radar`와 `card-mizer`로 백엔드 정체성을 먼저 잡는다.
2. `SignalMate`와 `blog-to-shorts`로 AI와 제품 확장 폭을 보여준다.
3. `Session Pilot`과 `discord-kakao-translator`로 실제 사용자 문제를 푸는 감각을 보강한다.

이 순서가 중요한 이유는, 포트폴리오가 "분산된 사이드 프로젝트 모음"으로 보이지 않고 "백엔드에서 출발해 제품 전체로 확장해 가는 개발자"의 흐름으로 읽히게 하기 위해서다.

## 대표 6개 프로젝트의 역할

### asset-radar

- 역할: 백엔드, 데이터, 분석, 인프라를 한 번에 묶는 대표 프로젝트
- 증명할 것: WebFlux, Kafka, Redis, PostgreSQL, 실시간 API, 대시보드, 모니터링
- 공개 강조점: 기술 나열보다 "실시간 자산 데이터 파이프라인과 분석 시스템"으로 설명

### card-mizer

- 역할: 구조 설계와 도메인 모델링 역량을 보여주는 백엔드 축
- 증명할 것: 헥사고날 아키텍처, 멀티모듈, 규칙 기반 추천, 테스트
- 공개 강조점: CRUD가 아니라 도메인 판단 로직 중심 백엔드라는 점

### SignalMate

- 역할: AI와 제품 기획, UX, 포지셔닝 감각을 보여주는 핵심 후보
- 증명할 것: 문제 정의, 랜딩, 분석 경험, API 프로토타입, 결제/전환 설계
- 공개 강조점: "연애 예언"이 아니라 "근거 중심 신호 해석과 다음 액션 제안"

### blog-to-shorts

- 역할: AI 자동화와 마케팅/콘텐츠 파이프라인을 보여주는 프로젝트
- 증명할 것: 크롤링, LLM 스크립트 생성, 미디어 생성, 업로드 자동화
- 공개 강조점: 기술 스택보다 "콘텐츠 운영 자동화"와 "성장 도구" 관점

### Session Pilot

- 역할: AI 시대의 새로운 workflow 문제를 다루는 제품 축
- 증명할 것: 자동 감지, 검토 큐, 재진입 요약, 작업 관제 UX
- 공개 강조점: "AI 채팅 앱"이 아니라 "AI 작업 인박스/관제판"

### discord-kakao-translator

- 역할: 작지만 분명한 사용자 문제를 끝까지 해결하는 실행력 증명
- 증명할 것: 플랫폼 연동, LLM 활용, 상태 UI, 트레이 앱, 실패 처리
- 공개 강조점: 일상적인 불편을 실제 동작하는 도구로 닫은 경험

## 공개면에서의 기본 원칙

- 과장하지 않는다. 구현 전 단계인 것은 `진행 중`, `프로토타입`, `구상 중`으로 명확히 적는다.
- 프로젝트 설명은 기술 목록보다 해결하려는 문제와 결과를 먼저 둔다.
- README, GitHub pin, 포트폴리오 페이지의 톤과 한 줄 설명을 최대한 맞춘다.
- 각 프로젝트는 "무엇을 증명하는 저장소인지"가 한 문장으로 설명돼야 한다.

## GitHub pin 구성 원칙

- `asset-radar`, `card-mizer`는 가능한 한 오래 유지하는 고정축으로 본다.
- 나머지 4칸은 포트폴리오 전체 메시지에 맞춰 유동적으로 조정한다.
- AI/product 영역 2개, workflow/user-problem 영역 2개 구성이 기본값이다.

## 대체 후보

### meetric

- 장점: 일반 사용자용 UI/UX 프로젝트로 읽기 쉽다.
- 쓰는 경우: AI 색을 조금 줄이고 더 대중적인 웹 제품 감각을 넣고 싶을 때

### webhook-notify

- 장점: SDK 설계와 라이브러리 배포 경험을 보여주기 좋다.
- 쓰는 경우: 제품형 프로젝트보다 플랫폼/라이브러리 역량을 강조하고 싶을 때

## 운영 메모

- 포트폴리오 페이지는 "보여주는 곳"이다.
- `docs/` 문서는 "다음에 어떻게 고도화할지"를 정리하는 곳이다.
- 저장소 README는 "각 프로젝트 단위의 설득"을 담당한다.
